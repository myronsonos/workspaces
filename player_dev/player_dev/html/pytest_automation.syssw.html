pytest_automation.syssw.conftest.pytest_addoption(parser)[source]¶

Add test specific command line options.

Since this conftest.py is common to all tests, the command line options are designed to be somewhat generic.

–iterations:
This parameter itself will not perform any tasks. To iterate your tests, include the parameter in your script (e.g. for loop) to perform the activities multiple times. If you want to parameterize a test case see ‘–repeat-times’.
–repeat-times:
This command line option will automatically collect each test and execute for predefined number of times. No need to include it in your test script. Default is 1, which means each test will only be executed once.
–keep_devices_in_error_state:
If specified, this option can be used by a test to keep a device in it’s error state in the event of a failed testcase. As an example, a test may handle this by rebooting devices, starting anacapa, or clearing errors from dmesg if this is not specified. This holds a bool and is ‘False’ when not specified and ‘True’ when specified.

pytest_automation.syssw.test_neptune_output_sync module¶

This test is to check that Neptune’s analog and digital outputs are properly synced. This test is unique to Neptune because other Connect-like products utilize a single sound card for both digital and analog outputs. Neptune is the first Sonos Connect-like player to use two separate sound cards for the outputs.

This test utilizes the clock-time in /all/syslib/tools/clock-time The app generally compares tx and rx, but it has been tweaked to accept spdif-tx as an “input” to compare with the analog tx.

Test cannot replace actual sync testbed that will be managed my the DSP team, which will be more reliable. This ensures that at least from the software’s perspective, all is good.

** NOTES ** This test is considered a full fail if ANY of the tests fail. Currently configured to fail on –maxfail=1 in the DSL job definition. Doesn’t have to run in order. Would be slightly more efficient but we wouldn’t get failure logging which could be useful. The time difference test outcome is useless if the parallel test fails

Questions/concerns: james.park@sonos.com or SYSSW QA
