upnp.functional.smapi_reporting.track_play_reporting module¶

This test suite covers the trackPlayReporting mechanism players use to report time played events to SMAPI partners. This includes the “setPlayedSeconds”, “reportPlaySeconds” and “reportPlayStatus” SMAPI calls. These tests were built off of the legacy behavior before SWPBL-36988 work was completed.

For more information on the expected behaviors for SMAPI track play reporting please see: https://confluence.sonos.com/x/4QG_Aw

class upnp.functional.smapi_reporting.track_play_reporting.TrackPlayReportingContextIdTests(fixture_name=None, *args, **kwargs)[source]¶

Bases: sonos.workflow.fixture.WorkflowTestFixture

These tests set up the node-smapi server on the household with the added capability to report the contextId along with the reportPlaySeconds and setPlayedSeconds calls. This feature was added in Mudhoney and is tested in a different class as it is only in use by Apple music at the time these tests were written (Sprint 3 2016).

The contextId returned is the parent container the tracks came from and is added to the report based on an entire container being added to the queue or played from the browse node. As such the method of adding tracks to the queue is the same but requires a different set of MD to be passed in. These tests are not exhaustive, but test the main happy path that when a container is enqueued we report the contextId and that when a track is played normally the player only reports the track itemId as the contextId.

See Yves comment on this page for more information (this feature is not yet documented anywhere officially): https://confluence.sonos.com/display/PDSW/SMAPI+Track+Reporting+Changes?focusedCommentId=73406001#comment-73406001

test_change_track_position_in_queue_while_playing()[source]¶

Verify the track play segments are coalesced correctly when the current playing track has its position in the Sonos queue changed by adding or removing a track before it. This tests any dependence our reporting has on the trackId (Sonos queue position ID) and how our string pool allocation works. Once the string pool hits 50 entries we stop collecting track information until that buffer can be cleared by reporting and the trackId is used in the segment reporting which means a queue position change can cause different segments to take up more entries than a normal play through.

Verify when a SMAPI server responds to the first reportPlaySeconds call with a new interval the player should send the reportPlaySeconds call on that new interval after the initial report.

NOTE: the requested iteration begins after the initial report. So an iteration change to 10 seconds would result in reports at 0, 10, 20, 30… seconds.

Verify when a SMAPI server responds to the first reportPlaySeconds call with a new interval of 0 the player never sends another reportPlaySeconds call for that track. Effectively this means only one report will be sent for the track (the start track report).

This was an implementation detail for the Mudhoney project with Apple music on SMAPI. See SWPBL-45109
