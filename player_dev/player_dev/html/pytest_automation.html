pytest_automation.conftest.capture_anacapa_logs(*args, **kwargs)[source]¶

Capture anacapa.trace logs from capture_anacapa_devices. Optionally scan incoming messages for strings specified by capture_anacapa_watch_strings.

Devices are restored to a normal state after the testcase completes.

Warning

Changes made by this fixture will be reverted if testbed devices are restarted/bounced unless write_anacapa_logs is set to True, in which case, we’ll still be able to watch the strings once the player begins sending us logs again after reboot/bounce.

Note

The following keyword arguments can be used to control the behavior of this fixture (when used on testcases, classes, or modules).

  • “capture_anacapa_fixture_names”
  • “watch_for_strings”
  • “write_anacapa_logs”

Refer to capture_anacapa_devices, capture_anacapa_watch_strings, and write_anacapa_logs fixtures for more details.

Note

To access captured anacapa.trace messages for a device, use get_udp_log().

To check if a str being watched for has been detected in incoming anacapa.trace logs, use has_search_results().

To get the number of times a str being watched for has been detected in incoming anacapa.trace logs, use get_search_results().

This fixture can be used either as a testcase/class.

Using ‘capture_anacapa_logs’ decorator on test case/class¶
@pytest.capture_anacapa_logs(
    capture_anacapa_fixture_names="playback_device",
    watch_for_strings=["large sync error", "small_sync_error"],
    should_write=True)

This fixture can be applied to all testcases in a module.

Using ‘capture_anacapa_logs’ on all testcases in module¶
pytestmark = [
    pytest.capture_anacapa_logs(
        capture_anacapa_fixture_names=["playback_device", "bridge"],

Pytest hook which implements rules for Pytest runlevels.

Runlevel filters are methods which are implemented by test writers to determine if a given parameter should be used by a testcase.

For example, a testcase “test_device” may use the device fixture, which is a parametrized fixture that returns each device found on the testbed.

  • At the low runlevel, only some specific platforms may run.
  • At the medium runlevel, only secureboot capable devices may run.
  • At the highest runlevel, all devices are used.

Filtering logic can be performed for individual testcases, for all testcases in a module, or for all tests found within a directory.

Filters for individual testcases are implemented by creating a method in the test module named “runlevel_<testcase name>”. For example, if you wanted to implement a runlevel method for the testcase “test_something”, you would name the method “runlevel_test_something”.

Filters applying to all testcases in a test module are implemented by creating a method named “runlevel” in the module.

Filters which should run against all testcases in a directory can be implemented by creating a method named “runlevel” in the local conftest.py file.

Filtering methods should return:
  • True if the parmaeter should be used.
  • False if the parameter should not be used.

Filtering can be performed on any type of parameter.

Note

A “runlevel_<testcase>” method will always take precedence over a module or conftest.py runlevel filter.

A “runlevel” method in a test module will always take precedence over a conftest.py runlevel filter.

A “runlevel” filter in a conftest.py file will apply to all testcases in a directory, unless more specific filters are defined.

Note

The ‘runlevel’ method (when implemented in a conftest.py file) is not treated as a Pytest hook. Instead, it is called directly by this plugin. This ensures that filtering logic implemented is applied only to the appropriate testcases.
