upnp.functional.queue.add_to_queue module¶

UPnP Functional Test: Queue.AddToQueue Tests the usage of the AVTransport:AddURIToQueue and AVTransport:AddMultipleURIsToQueue actions

These tests do NOT use the ‘EnqueueAsNext’ feature, which is used with SHUFFLE. (i.e. when SHUFFLE mode is enabled, setting ‘EnqueueAsNext’ to true will force the “next playing track” to be a track added. When SHUFFLE is not enabled, the flag has no effect)

upnp.functional.queue.check_metadata module¶

UPnP Functional Test: Queue.CheckQueueMetadata Tests the eventing of queue metadata for various track/albums

There are a few ways of getting the raw data used in this test. Here’s one method:
  1. Using Sonos CR, queue up the things you want
  2. Using DeviceSpy, Seek() to the various tracks (if an album, just one of the tracks)
  3. Copy the evented EnqueuedTransportURI and EnqueuedTransportMetaData
  4. Use those values in the test case
  5. Run the test fixture, populate (and visually check) the track data into the test case

test_private_queue_stringtable_increase()[source]¶

This script isn’t really a test, it’s a helper to verify the P2S private queue string table increase made for PTS-841. It was easier to write it this way quickly. This will add 99 tracks to the private queue with 4k metadata for each and add enough tracks to almost meet the 65k metadata limit on the shared queue. There is a massive sleep added because the test must remain running in order for playback to continue - when the script ends the webserver goes away. If you let this run it will also submit a diag at the end. Failure to add to the public or private queue are caught as exceptions.
